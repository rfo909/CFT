4.0.0 2023-02-02

And there, it just felt like version four, with interactive OS commands integrated into CFT without special syntax.

Looking through the list of what has been done since 3.0.0 and roughly comparing to the work between v2 and v3, it is clear that it is time for version 4.

Even as the 3.8.3 change of running OS commands is a bit hack-like, it works well. May at some point decide to formalize the detection of where errors come from, to differentiate between an input line meant for the OS, and an input line with bad but attempted CFT syntax.

So ... will there be a version 5??

My private TODO list has shrunk quite a bit lately. Some of the remaining (big) ideas follow.

Pluggable functionality

Formalizing an interface or a way of adding pluggable functionality, using CFT as control script for external and specialized processes that do not naturally fall into the CFT code tree.

Have even considered writing terminal games ...

I/O and terminal handling

Another advanced issue is improved I/O and terminal handling, so that we can create some rule set for automatic submitting commands not only to a regular external program, but also to ssh, so that CFT can login automatically with username + password over ssh, if needed. This is a big "can of worms" compared to the simplistic one-line-at-a-time I/O model, which has worked very well so far.

Avoid blocking or hanging commands

One application of the above would be a long-dreamed solution to the problem of how both CFT code and in particular external commands sometimes will take way too long, even hanging. The idea here is that all commands issued at the command line are run in a separate process, and the REPL remains attached to the most recent, interacting with it normally, until we press Esc or something, at which point we disconnect, leaving it running in the background, and returns us to prompt.

Sub-languages

The original plans, before deciding on functions, was that every (or some) objects could support their own mini language, parsed separately, perhaps only depending on respecting the use of curly braces. One such language could be multiline OS native scripts, corresponding to how we use OS commands directly from the interactive command line ... 

	Std.Ext {
		cd (WorkDir)
        mvn clean package
        cp target/*packages*.jar (TargetDir)
	}

OS commands inline in code

Could also let the "bang" notation be supported inline in CFT code. 

	WorkDir.cd
	!mvn clean package
	!cp target/*packages*.jar (TargetDir)

Networking

A long-time concept has been that of letting instances of CFT communicate with each other, so that they could form a network, allowing us to manage hosts only available indirectly, via jump hosts, from the local PC. This would also really require running CFT as services, but the main reason it has never surfaced for real, are the security concerns. I mean, bypassing ssh or windows logins? Not likely to be accepted in any real world environments.

I did a test on invoking CFT commands remotely in a separate CFT process some time during 2021. It worked, but had no real use by itself, and was deleted from the code base. The java class StdioVirtual is a remnant from this test, which later has seen use for managing I/O for background jobs and processes.

3.5.3 2022-12-01 Stored symbols

After deciding CFT needs "destructive" shell-like commands, and wondering how to implement cp, mv and diff in an efficient way, particularly when spanning different directories, and also how to jump between various directories and files, the Symbols notation was invented.

This consists of the following parts:

  • mSymGet and mSymSet lambdas defined in CFT.props, pointing to Lib:SymbolGet and Lib:SymbolSet
  • New expression ExprSymGetSet parsed via ExprTerminal, on format %%xxx to set and %xxx to read symbol value
  • ExprSymGetSet calls the two new lambdas. The Set stores the LastValue in Db2, and Get reads it back
  • Added a shortcut @% that lists and lets us delete symbols
cd /some/project
pwd
%%proj

# ... later: go to that directory
%proj.cd 

# ... or list files 
%proj.files

Both %x and %%x are valid expressions, and in a sense they are global variables, but intended for interactive use, and tied in with Sys.lastResult. Ideally we could have implemented %%x parsing inside the interactive command line interpreter instead of as a general language feature, to make that point clearer, but %%x is pretty useless inside general code, as it will always return Sys.lastResult, which is fixed until next command line input is completed.

Also added warning when trying to define symbol value that is neither File nor Dir, as thats what the mechanism is mostly assumed to be used for, jumping between locations, comparing and copying between locations, etc.

3.5.0 2022-04-22 Classes version 2

Changed classes from v3.3.2 in-function auto-inherit of function parameters, returning to a modified function definition: 

	# Class A
	# --
		P(1)=>x
		self.x=x
		...
	/class A

In both the old and current implementation, the aim is to generate a "class object", which can be used directly as an object, but also can be instantiated and copied into an ohter class object, which then becomes a sub-class of the first.

The code above defines classes in the same way as functions, but instead of /name, we use the /class ... line, which comes in two variants: 

	/class A
	/class A as T

The A is the name of the function, which creates the class object, which is the one we call. The optional part, T, is used to define the class object type. If no type is given, the type defaults to the class function name.

The type of an object in CFT is kind of a "promise" or perhaps "an informal interface name", which is selected freely, to simplify functions and lambdas doing "as" type checking, not having to know all subclasses of some class, but instead either use a root class name or some other freely selected name to describe a set of features, and check for that.

The OODemo script illustrates all this.

3.4.4 2022-04-18 Lambdas are not synthesizable

Currently Lambdas are anonymous functions that are defined within a script environment, and like other functions, have access to functions in that environment directly by name. This means we can not synthesize them and bring them back in code inside another script and expect them to work.

Behind the scenes, the class expression sets up the self dictionary, then runs the block of code as a lambda, with "self" as a local variable, and inheriting parameters from the function, and finally, returns the self object. 

# Test class
# --
	class {
		P(1) as String => name
		self.name=name
		self.present=Lambda{
			"<b>"+self.name+"</b>"
		}
	}
/TestClass

# Example use
# --
	tc=TestClass("this is a test")
	tc.present
/example

Inheritance

Inheritance uses the fact that "self" is a dictionary, and has a .copyFrom(otherDict) function. This means that there is no path up the inheritance chain, but instead that we copy content from super-class to subclass.

It is all about creating objects that reuse code. As we see below, the subclass updates its' self with content from super class ("TestClass"), then copies the reference to self.present into self.origPresent before redefining self.present, which in turn calls self.origPresent. This of course corresponds to "super.present()" in Java, but is more manual. There also is nothing preventing multiple inheritance, as the content of the "self" is managed programmatically in the class block. 

# Subclass
# --
	class {
		P(1) as String => name
		
		# inheritance via copying object content
		self.copyFrom(TestClass(name))

		self.origPresent=self.present
		self.present=Lambda{
			# Calls inherited (copied) render function from TestClass
			@ <p>Extra text</p><< self.origPresent >>
			.mergeExpr
		}
	}
/Subclass

# Example use
# --
	Subclass("test").present
/example2

2.12.1 2021-12-16 Background jobs

Created alternative notation for the SpawnProcess() expression: 

   & expr
   & expr, name

The name is either an identifier or an expression, typically a string. This extends the default name, which is "(n)", for readability.

Updated the _Expr global documentation function.

Also modified the Sys.Jobs and the Jobs script from yesterday, as well as the shortcuts defined in CFT.props. Developed a crude way of bringing a running job to the foreground, so that jobs stalled on input, can be supplied with input data.

Some details

As the Input object is unaware of threads, the "last value" functionality becomes unreliable if there are more than one job running the same code, because one may update the "last value" after another has blocked on the same input, telling the user what was the "last value" at that point, causing confusion.

The rapid development of background jobs was driven by a very specific need at work, with slow servers and tens of thousands of log files that needed to be "massaged" into meaningful subsets, stored in Db2, for further inspection. The problem was that searching the remote directories for the latest file for some Glob pattern, was way too slow to repeat for every inspection. I wrote some "update"-functions which extracted subsets of files, and stored them in the database, and now these functions can be run in the background. Very nice.

Inside code

Even though the '&' expression (ExprAmp internally) is runnable from general code, it is limited compared to SpawnProcess, as it can not refer variables. On the command line that is not a problem, since we don't have any variables around anyway. Also there is the thing that jobs created with ExprAmp are added to the internal registry, and returns the job name.

In code we will normally use SpawnProcess, which returns an ObjProcess object.

Still, any code can easily use ExprAmp, when we want to automate generating background jobs for further management via the Jobs script (which in turn uses the Sys.Jobs object).

Also deleted the Cron script from 2.9.10, as it adds nothing of value any more.

v1.3.3 2020-09-14 Lib.Text.Parser

Have by now mostly figured out how the Parser should behave. For CFT we process the complete script file (or input line) into tokens first, then run recursive-descent parsing, coded in Java, with objects creating objects recursively. The CFT parser uses a lookahead of 1, and does not backtrack (the Lexer does).

Creating a generic parser that will be used for both structured data such as JSON or HTML, and processing log-lines with potentially complex and overlapping token definitions, this parser will need to do backtracking. This means lookahead none, supporting multiple productions per non-terminal symbol.

The "productions" scheme for grammar representation is inspired by EBNF, which is the Extended Backus-Naur Form. which evidently extended BNF by allowing '?', '*' and '+' modifiers in the grammar, saving lots of problematic (potentially infinite) recursion.

I've made a further extension in the same direction, allowing not only the above modifiers, but for '*' and '+' also allowing reference to a second separator non-terminal that has to be matched inbetween repeating instances of the first non-terminal. It complicates the implementation a bit, but greatly simplifies the grammar.

2020-06-14 Macro

Created ExprMacro which is { Stmt* } and which when executed returns a ValueMacro object. This is a regular value, though not synthesizable, which can be stored in a variable or returned from a function,

To call a macro, invoke the .call function on it, possibly with parameters, which are in turn picked up with the usual P() function.

Macros come in two variants: local and global. A local macro has access to variables in the caller, and will redefine those when assigning, rather than create internal local variables. These macros are for internal use inside functions. This means that local variables inside the macro should be limited, as they have to be unique to avoid confusion. 

	{ <- .... }

Global macros are scope-isolated macros that do not inherit anything from the caller environment. They can be passed around as parameters, and called by anyone, without unfortunate side effects or errors due to missing symbols. 

	{ .... }

Currently macros do not support loops or the PIPE character.

2019-03-22 syn() function

In preparation for creating multiline scripts, as well as for general flexibility, I've added the global syn() function, which takes a synthesizable value, and returns as a string the code that creates it. If the value can not be synthesized, an error is reported.

Also added String.indexOf() function for improved parsing.

2019-02-17 Function.java

(2019-03-13 updated)

As part of creating a Gui class which should contain interactivity, we needed to extend the callFunction() of the Function class, as invoked from LookupOrCall.resolve(). Created a parallel method, called callFunctionNew(), which by default calls the callFunction() method, which is now defined in Function.java throwing exception.

The callFunctionNew() takes Ctx object instead of TextIn and TextOut, which are anyway direct properties of the Ctx object.

Also extended the CommandProcessor interface to include the method processProgramLine.

In sum this makes it possible to create code that executes code. The first example is the Gui.button("name","code"), which when having its "clicked" function called, executes the "code", using the CommandProcessor.processProgramLine() method. This method returns a value, which becomes the return value from the clicked() method. This of course exemplifies how an eval("str") function should work.

So obviously eval(str) was the next to be created.

2019-02-17 ObjPersistent

After creating Gui.show, which displays a graphical interface, a method for killing that window on exit was needed.

This was done by extending the ObjPersistent class with a method cleanupOnExit(), which for ObjGui performs a dispose on the App object. Also added identically named method to ObjGlobal, and a call to that from Main on exit.

Also added method initPersistentObj() which is called once for each persisted instance. Added this method to the Gui object (ObjGui), as a way of popping up the JFrame after it has been created and recognized as the first Gui object. Later calls to Gui which returns objects with persistence-id matching an existing object are thrown away, and replaced by the persisted object.

2019-03-13 showCode()

Added wildcard '*' option at end of name matches.

Example: showCode("DirCurr,Dir*:Config|S*:Search|*:The rest")

The point of DirCurr before Dir* is to ensure DirCurr is displayed before other Dir* whatever their alphabetical order.
